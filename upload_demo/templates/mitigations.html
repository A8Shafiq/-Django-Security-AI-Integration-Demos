SQL Injection Mitigations

Parameterized Queries (Prepared Statements) CRITICAL

Never concatenate user input into SQL strings. Use parameterized queries where the database engine treats inputs as data, never as executable code.

Why it works:

The database separates the query structure from the data. Even if the input contains ' OR 1=1 --, it's treated as a literal string value, not SQL syntax.

❌ Vulnerable (string concatenation):

cursor.execute("SELECT * FROM users WHERE username = '" + username + "'")

✅ Secure (parameterized):

cursor.execute("SELECT * FROM users WHERE username = %s", [username])

Use an ORM (Object-Relational Mapping) HIGH

Use Django ORM, SQLAlchemy, or similar frameworks that automatically build parameterized queries. Avoid raw() and extra() unless absolutely necessary.

Why it works:

ORMs abstract away raw SQL entirely. User.objects.filter(username=input) always generates a safe parameterized query internally.

✅ Secure (Django ORM):

user = User.objects.filter(username=username).first()

Input Validation & Whitelisting MEDIUM

Validate that inputs conform to expected formats (e.g., alphanumeric usernames, numeric IDs). Reject unexpected characters at the application layer.

Why it works:

Defense-in-depth: even if parameterization is missed somewhere, validation reduces the attack surface by stripping SQL metacharacters.

✅ Example validation:

import re if not re.match(r'^[a-zA-Z0-9_]{3,30}$', username): raise ValidationError("Invalid username format")

Principle of Least Privilege MEDIUM

Database accounts used by the application should have minimal permissions. A login form only needs SELECT on the users table — not DROP, ALTER, or access to other tables.

Why it works:

Even if injection succeeds, the damage is limited. An attacker can't DROP TABLE if the DB user doesn't have that permission.

Cross-Site Scripting (XSS) Mitigations

Output Encoding / Auto-Escaping CRITICAL

Always escape user-generated content before rendering it in HTML. Django's template engine auto-escapes by default — never use the |safe filter on untrusted data.

Why it works:

Escaping converts <script> to &lt;script&gt;. The browser renders it as visible text instead of executing it as code.

❌ Vulnerable (|safe filter):

{{ "{{ comment.text|safe }}" }}

✅ Secure (auto-escaped):

{{ "{{ comment.text }}" }} {# Django auto-escapes this #}

Content Security Policy (CSP) HIGH

Set CSP headers to restrict which scripts can execute. A strict CSP prevents inline scripts and only allows scripts from trusted origins.

Why it works:

Even if an attacker injects <script>alert(1)</script>, the browser refuses to execute inline scripts when CSP disallows them.

✅ Example CSP header:

Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'

HttpOnly & Secure Cookie Flags HIGH

Set cookies (especially session cookies) with HttpOnly and Secure flags. This prevents JavaScript from accessing them.

Why it works:

HttpOnly makes document.cookie return empty for that cookie. Even successful XSS can't steal session tokens.

✅ Django settings:

SESSION_COOKIE_HTTPONLY = True SESSION_COOKIE_SECURE = True CSRF_COOKIE_HTTPONLY = True

DOM-Safe APIs HIGH

In JavaScript, use textContent instead of innerHTML. Never pass user input to eval(), setTimeout(string), or document.write().

Why it works:

textContent treats everything as plain text — HTML tags are not parsed or executed by the browser.

❌ Vulnerable:

element.innerHTML = userInput; // Parses HTML, executes scripts

✅ Secure:

element.textContent = userInput; // Plain text only

File Upload Mitigations

Extension Whitelist CRITICAL

Only allow specific, known-safe file extensions. Use a strict whitelist — never a blacklist (attackers always find extensions you didn't think of).

Why it works:

Blocking .php is a blacklist — attacker tries .phtml, .php5, .phar. Allowing only .jpg, .png, .pdf is a whitelist — attacker has no bypass.

✅ Example:

ALLOWED = {'.jpg', '.jpeg', '.png', '.gif', '.pdf'} ext = os.path.splitext(filename)[1].lower() if ext not in ALLOWED: raise ValidationError("File type not allowed")

MIME Type + Magic Bytes Validation HIGH

Check both the Content-Type header AND the file's actual content (magic bytes / file signature). The python-magic library can detect the real file type regardless of extension.

Why it works:

MIME headers can be spoofed, but magic bytes are embedded in the file's binary data. A PHP script pretending to be JPEG won't have the FF D8 FF JPEG signature.

✅ Example with python-magic:

import magic mime = magic.from_buffer(file.read(2048), mime=True) if mime not in {'image/jpeg', 'image/png', 'application/pdf'}: raise ValidationError(f"Invalid file content: {mime}")

File Size Limits HIGH

Enforce maximum file size at both the application level and the web server level (nginx/Apache). This prevents disk exhaustion and zip-bomb attacks.

Why it works:

Limiting to e.g. 2 MB makes DoS impractical — attacker would need millions of requests to fill the disk, and rate limiting stops that too.

✅ Django settings:

# settings.py DATA_UPLOAD_MAX_MEMORY_SIZE = 2 * 1024 * 1024 # 2 MB FILE_UPLOAD_MAX_MEMORY_SIZE = 2 * 1024 * 1024

Rename Files on Upload HIGH

Never use the original filename. Generate a random name (UUID) and keep the whitelisted extension. Store the original name in the database if needed for display.

Why it works:

Eliminates path traversal (../../etc/passwd), prevents file overwrites, and stops information leakage from predictable filenames.

✅ Example:

import uuid, os safe_name = f"{uuid.uuid4().hex}{ext}" save_path = os.path.join(UPLOAD_DIR, safe_name)

Store Outside Web Root CRITICAL

Never store uploaded files in a directory served directly by the web server. Use a separate storage location and serve files through an application-controlled endpoint with proper access checks.

Why it works:

If a PHP file is stored outside the web root, Apache/Nginx won't process it as PHP when accessed via URL — it's just data, not executable.

Antivirus / Content Scanning MEDIUM

Scan uploaded files with ClamAV or similar tools before accepting them. This catches known malware, trojans, and malicious scripts embedded in files.

Why it works:

Signature-based detection catches known threats. Combined with other mitigations, it provides an additional defense layer against zero-day exploits.

General Security Best Practices

Defense in Depth

Never rely on a single security measure. Layer multiple controls so that if one fails, others still protect the application. For example, combine input validation + parameterized queries + least-privilege database accounts for SQL injection defense.

Security Headers

Set HTTP security headers on all responses to protect against common attacks:

X-Content-Type-Options: nosniff # Prevents MIME-type sniffing X-Frame-Options: DENY # Prevents clickjacking X-XSS-Protection: 1; mode=block # Legacy XSS filter Strict-Transport-Security: max-age=... # Forces HTTPS Referrer-Policy: strict-origin # Controls referer leakage

Keep Dependencies Updated

Regularly update frameworks, libraries, and server software. Use tools like pip-audit, npm audit, or Snyk to scan for known vulnerabilities in dependencies.

Logging & Monitoring

Log all authentication attempts, file uploads, and unusual queries. Set up alerts for patterns like repeated failed logins, SQLi payloads in logs, or large file uploads. Use a SIEM (Security Information and Event Management) in production.

Web Application Firewall (WAF)

Deploy a WAF (e.g., ModSecurity, AWS WAF, Cloudflare) as an additional layer. WAFs can detect and block common attack patterns before they reach your application code.
