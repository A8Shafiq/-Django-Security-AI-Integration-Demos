📁 File Upload Vulnerability Lab

File upload is a common attack vector. If an application accepts user-uploaded files without proper validation, attackers can upload malicious content to compromise the server, steal data, or attack other users.

🔥 Exploits & How They Work

🐚

Web Shell Upload

Attacker uploads a server-side script (PHP, Python, JSP) that gives them remote command execution on the server.

  1. Attacker creates a file named shell.php containing <?php system($_GET['cmd']); ?>
  2. Uploads it through the vulnerable form — server saves it to /uploads/shell.php
  3. Attacker visits /uploads/shell.php?cmd=whoami in the browser
  4. The web server executes the PHP file — the attacker now has full shell access
CRITICAL — Remote Code Execution
📂

Path Traversal via Filename

Attacker manipulates the filename to write files outside the upload directory, overwriting system or config files.

  1. Attacker crafts a filename like ../../etc/cron.d/backdoor
  2. Server blindly uses the original filename when saving
  3. The ../ sequences navigate up directories
  4. The file is written to a system directory — attacker gains persistence
HIGH — Arbitrary File Write
🎭

Double Extension Bypass

Attacker uses a filename like image.php.jpg to bypass naive extension checks that only look at the last extension.

  1. Server checks only the last extension → sees .jpg → allows upload
  2. Some web servers (e.g. Apache with misconfigured handlers) process files left-to-right
  3. Apache sees .php and executes the file as PHP
  4. Attacker achieves code execution despite the "image" extension
HIGH — Extension Bypass
🕵️

MIME Type Spoofing

Attacker changes the Content-Type header in the HTTP request to make a malicious file appear legitimate.

  1. Attacker uploads shell.php but intercepts the request (using Burp Suite)
  2. Changes Content-Type from application/x-php to image/jpeg
  3. Server checks only MIME type → sees image/jpeg → allows it
  4. The file is still PHP — server executes it on access
MEDIUM — Validation Bypass
⚡

Stored XSS via File Upload

Attacker uploads an SVG or HTML file containing embedded JavaScript that executes when any user views it.

  1. Attacker creates an SVG: <svg onload="alert(document.cookie)">
  2. Uploads it — server saves as /uploads/image.svg
  3. When another user opens the SVG URL in their browser, the JS runs
  4. Attacker steals session cookies, redirects to phishing pages, etc.
HIGH — Cross-Site Scripting
💣

Denial of Service (DoS)

Attacker uploads extremely large files or many files to exhaust server disk space, bandwidth, or processing power.

  1. Attacker uploads a 10 GB file — no size limit exists
  2. Server disk fills up → other services start failing
  3. Or: attacker uploads a "zip bomb" that expands to terabytes when decompressed
  4. Server crashes or becomes unresponsive — legitimate users are locked out
MEDIUM — Availability Impact
